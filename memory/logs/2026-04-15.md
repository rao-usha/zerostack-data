## 15:45 ‚Äî Fix: ingestion_jobs.data_origin column missing on Cloud SQL

**What:**
- `/api/v1/jobs?source=synthetic_lp_gp` was 500ing: `column ingestion_jobs.data_origin does not exist`
- Column was defined in the model (`app/core/models.py:110`) and in the auto-migration list (`app/core/database.py:86`) as part of PLAN_053/056/057 synthetic-data work, but had never been applied against Cloud SQL
- Root cause #1: `ingestion_jobs` in Cloud SQL is owned by `postgres`; the app's `nexdata` role lacks ALTER privilege, so the `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` at startup raised `InsufficientPrivilege`
- Root cause #2: `_apply_schema_migrations` swallowed that error at `logger.debug` level, so the schema drift was invisible in logs ‚Äî API booted "healthy" while model/DB were out of sync

**Fix:**
- Reset Cloud SQL `postgres` password via `gcloud sql users set-password` (user did this from Git Bash + winpty)
- Connected as `postgres` through the existing `nexdata-cloudsqlproxy-1` container and ran:
  - `ALTER TABLE ingestion_jobs ADD COLUMN IF NOT EXISTS data_origin VARCHAR(16) NOT NULL DEFAULT 'real'`
  - `ALTER TABLE ingestion_jobs OWNER TO nexdata` (so future auto-migrations apply without superuser)
- Upgraded `app/core/database.py:94` from `logger.debug` ‚Üí `logger.error` so any future migration failure is loud

**Files changed:**
- `app/core/database.py` (migration error log level)

**Verified:**
- `data_origin` column present (default 'real')
- Table owner is now `nexdata`
- `GET /api/v1/jobs?source=synthetic_lp_gp` ‚Üí HTTP 200 `[]`

**Lessons / rubric updates:**
- Any Cloud SQL table created by the superuser will block app-user ALTERs. When adding schema migrations, either (a) ensure the target table is owned by `nexdata`, or (b) document that the migration needs a one-time admin run.
- Never log migration failures at DEBUG. Startup-time schema ops must surface loudly.

## 16:15 ‚Äî Perf fix: /api/v1/sources/health-summary (17.6s ‚Üí 0.7s)

**What:** Classic N+1. For 51 sources, the loop called 4 per-source helpers (`_health_job_info`, `_health_quality_scores`, `_health_schedule`, `_health_open_anomaly_count`) = 204 round-trips through the Cloud SQL proxy at ~85ms each.

**Fix:** Added four batched variants in `app/api/v1/sources.py`:
- `_health_job_info_all` ‚Äî single query with `ROW_NUMBER() OVER (PARTITION BY source ORDER BY completed_at DESC)`, bucketed rn<=2
- `_health_quality_scores_all` ‚Äî `SELECT DISTINCT ON (source) ... ORDER BY source, snapshot_at DESC`
- `_health_schedule_all` ‚Äî `DISTINCT ON (source)` where is_active=1
- `_health_open_anomaly_counts_all` ‚Äî one pull of all open alerts; bucket by longest-prefix match in Python (longest-first so `pe_collection` isn't also credited to `pe`)

Loop body now pure dict lookups. Per-source helpers left in place for other callers.

**Verified:** 17.6s ‚Üí 0.7s (25√ó speedup). Payload bytes identical (20625). Card shape unchanged (16 keys). Sample card for `fred` returns total_rows=20292, open_anomalies=3, row_trend=down as expected.

**Files changed:** `app/api/v1/sources.py` (added ~100 lines of batched helpers, 4-line change to the loop)

**Lesson:** When an endpoint that iterates a registry is slow, inspect for per-item DB calls first. Cloud SQL proxy round-trip ‚âà 80-100ms per query, so a 50-item loop with 4 helpers hits ~17s latency before any actual work.
