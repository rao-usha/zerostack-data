## 15:45 — Fix: ingestion_jobs.data_origin column missing on Cloud SQL

**What:**
- `/api/v1/jobs?source=synthetic_lp_gp` was 500ing: `column ingestion_jobs.data_origin does not exist`
- Column was defined in the model (`app/core/models.py:110`) and in the auto-migration list (`app/core/database.py:86`) as part of PLAN_053/056/057 synthetic-data work, but had never been applied against Cloud SQL
- Root cause #1: `ingestion_jobs` in Cloud SQL is owned by `postgres`; the app's `nexdata` role lacks ALTER privilege, so the `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` at startup raised `InsufficientPrivilege`
- Root cause #2: `_apply_schema_migrations` swallowed that error at `logger.debug` level, so the schema drift was invisible in logs — API booted "healthy" while model/DB were out of sync

**Fix:**
- Reset Cloud SQL `postgres` password via `gcloud sql users set-password` (user did this from Git Bash + winpty)
- Connected as `postgres` through the existing `nexdata-cloudsqlproxy-1` container and ran:
  - `ALTER TABLE ingestion_jobs ADD COLUMN IF NOT EXISTS data_origin VARCHAR(16) NOT NULL DEFAULT 'real'`
  - `ALTER TABLE ingestion_jobs OWNER TO nexdata` (so future auto-migrations apply without superuser)
- Upgraded `app/core/database.py:94` from `logger.debug` → `logger.error` so any future migration failure is loud

**Files changed:**
- `app/core/database.py` (migration error log level)

**Verified:**
- `data_origin` column present (default 'real')
- Table owner is now `nexdata`
- `GET /api/v1/jobs?source=synthetic_lp_gp` → HTTP 200 `[]`

**Lessons / rubric updates:**
- Any Cloud SQL table created by the superuser will block app-user ALTERs. When adding schema migrations, either (a) ensure the target table is owned by `nexdata`, or (b) document that the migration needs a one-time admin run.
- Never log migration failures at DEBUG. Startup-time schema ops must surface loudly.
